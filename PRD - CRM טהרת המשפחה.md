## 1. רקע ומטרה

### 1.1 רקע
עמותה הפועלת בתחום טהרת המשפחה מקיימת אירועים לנשים ברחבי הארץ. כיום, ניהול הלידים, האירועים, התרומות והמלאי מתנהל בכלים נפרדים (קבצי אקסל, WhatsApp, טפסי Google) ללא מערכת מרכזית.

### 1.2 בעיה
- מידע מפוזר בין כלים שונים → חוסר שקיפות ואיבוד לידים
- אין מעקב שיטתי אחרי לקוחות חוזרות וחד-פעמיות
- לא ניתן למדוד ROI לאירוע (תרומות מול הוצאות)
- תהליך הרישום לאירועים ידני וזמן-רב
- 3,000+ לידים בחודש – קיבולת גבוהה ממה שמוקד של 5 טלפניות יכול לנהל ידנית

### 1.3 מטרת המוצר
מערכת CRM ייעודית המאחדת ניהול לידים, אירועים, תרומות, מלאי והוצאות – תחת ממשק אחד עם הרשאות מדויקות לכל תפקיד.

### 1.4 מה המערכת לא עושה (Out of Scope - גרסה 1)
- גביית תשלומים דיגיטלית (רק תיעוד)
- ניהול הנהלת חשבונות מלאה
- אפליקציית iOS/Android נייטיב (PWA בלבד)
- CRM רב-ארגוני

## 2. פרסונות משתמשים

---

### פרסונה 1 – מנהל המערכת (Admin)
**שם לדוגמה:** יוסי, מנהל תפעול העמותה  
**גיל:** 40  
**טכנולוגיה:** משתמש מנוסה בכלים דיגיטליים

**אחריות:**
- אחראי כולל על פעילות העמותה
- מקבל דוחות ומדדים כספיים
- מגדיר מבנה המערכת (משתמשים, הרשאות, קטגוריות)

**צרכים מהמערכת:**
- לראות בזמן אמת: כמה נרשמו, הגיעו, תרמו ובכמה
- לנתח רווחיות לפי אירוע (תרומות פחות הוצאות)
- לנהל משתמשים ולהגדיר הרשאות
- לראות ביצועי מוקד (ממוצע לידים לטלפנית, שיעורי המרה)

**כאבים כיום:**
- קשה לאחד נתונים מגיליונות אקסל שונים
- לא יודע באיזה אירוע כדאי להשקיע יותר

**ציטוט מייצג:**
> "אני צריך לדעת עד כמה כל אירוע משתלם, לא רק כמה אנשים הגיעו"

### פרסונה 2 – מנהל/ת שיווק
**שם לדוגמה:** רחל, אחראית שיווק דיגיטלי  
**גיל:** 30  
**טכנולוגיה:** עובדת עם Meta Business, Canva

**אחריות:**
- מפרסמת קמפיינים בפייסבוק, אינסטגרם
- יוצרת דפי נחיתה לאירועים
- מנתחת אילו קמפיינים מביאים לידים איכותיים

**צרכים מהמערכת:**
- לבנות דפי נחיתה בקלות ללא מפתח
- לראות מאיזה קמפיין הגיע כל ליד
- לעקוב אחרי אחוזי הגעה ותרומה לפי מקור ליד
- לא לראות נתוני כסף

**כאבים כיום:**
- בונה דפי נחיתה בכלים חיצוניים ואחר כך מעבירה נתונים ידנית
- לא יודעת אם הלידים מפייסבוק "שווים" יותר מדף הנחיתה

**ציטוט מייצג:**
> "אני רוצה לבנות דף נחיתה ב-5 דקות ולראות מיד כמה נרשמו"

**טכנולוגיה:** נוחה עם מחשב, מעדיפה ממשק פשוט

**אחריות:**
- מתקשרת ללידים שהוקצו לה
- מזמינה לאירועים, מחזקת קשר עם לקוחות

**צרכים מהמערכת:**

- לדעת אם הלקוחה הגיעה לאירועים בעבר / תרמה

- לתזמן שיחת follow-up

**כאבים כיום:**
- לא יודעת מתי האחרונה דיברו עם הלקוחה
- צריכה לבדוק בגיליון אקסל נפרד אם הלקוחה הייתה באירוע

**ציטוט מייצג:**
> "פעמים רבות אני מתקשרת לאישה שכבר תרמה וזה עושה רושם לא טוב"

## 4. תהליכי עבודה מרכזיים

---

### תהליך A – מחזור חיי ליד (Lead Lifecycle)

```
[מקור דיגיטלי] → [ייבוא אוטומטי למערכת] → [חלוקה לטלפנית]
      ↓
[טלפנית מתקשרת] → [תיעוד שיחה + סטטוס]
      ↓
[שיוך לאירוע מתאים] → [רישום לאירוע] → [שליחת כרטיס WhatsApp]
      ↓
[הגעה לאירוע] → [סריקה בקבלה] → [תרומה / לא תרמה]
      ↓
[שמירה בהיסטוריה] → [ניתן לפנות לאירוע הבא]
```

**סטטוסי ליד:**
1. `ליד חדש` – נכנס, טרם טופל
2. `מוקצה` – הוקצה לטלפנית, טרם התקשרה
3. `בניסיון` – ניסו להתקשר, לא ענה
4. `בטיפול` – שיחה נפתחה, ממשיך מעקב
5. `מעוניינת` – הביעה עניין, ממתינה לאירוע
6. `נרשמה לאירוע` – רשומה לאירוע קרוב
7. `לא רלוונטי` – סירבה / מספר שגוי / לא מתאים
8. `לקוחה פעילה` – הגיעה לפחות לאירוע אחד

### תהליך B – יצירת אירוע ופרסום

```
מנהל שיווק יוצר אירוע ← הגדרת פרטים (שם, תאריך, מיקום, קיבולת)
      ↓
בניית דף נחיתה (עורך ויזואלי) ← תמונה, טקסט, שדות רישום
      ↓
פרסום דף נחיתה ← URL ייחודי
      ↓
שיתוף ב-Meta (קמפיין) ← לידים נכנסים אוטומטית
```

### תהליך C – רישום לאירוע *(מעודכן בגרסה 1.3)*

> **שינוי מרכזי:** כרטיס ה-QR **אינו נשלח** ברישום. הוא נשלח רק לאחר אישור הגעה טלפוני (תהליך J).

```
נרשמת מגיעה לדף נחיתה
      ↓
ממלאת שם + טלפון + עיר [+ שדות נוספים]
      ↓
בדיקת כפילות:
  ├── טלפון קיים + רשומה לאירוע הזה → "כבר רשומה" (ללא שליחה חוזרת)
  ├── טלפון קיים + לא רשומה → שיוך לרשומה קיימת + רישום לאירוע
  └── טלפון חדש → יצירת רשומת לקוחה + רישום לאירוע
      ↓
סטטוס אישור הגעה נקבע: `ממתינה לאישור`
      ↓
שליחת WhatsApp: הודעת אישור רישום בלבד ← ללא כרטיס/QR
  "תודה על הרישום! נציגה שלנו תצור קשר לאישור הגעתך"
      ↓
עמוד תודה בדף הנחיתה
      ↓
[המשך בתהליך J – אישור הגעה טלפוני]
      ↓
לאחר אישור הגעה בטלפון (הטלפנית מסמנת "אישרה הגעה"):
      ↓
יצירת ברקוד QR ייחודי (Event_ID + Contact_ID)
      ↓
שליחת WhatsApp: כרטיס דיגיטלי עם QR
```

### תהליך D – יום האירוע

### תהליך J – ניהול אישורי הגעה *(חדש בגרסה 1.3)*

**מטרה:** לוודא הגעה לפני האירוע ולהפיק כרטיס QR רק לנרשמות שאישרו; להתריע למנהל שיווק כשאחוז האישורים נמוך מהיעד.

**סטטוסי אישור הגעה (על EventRegistration):**

| סטטוס | משמעות |
|---|---|
| `ממתינה לאישור` | נרשמה, טרם הותקשר אליה |
| `ניסיון_1_ללא_מענה` | התקשרו פעם 1, לא ענתה |
| `ניסיון_2_ללא_מענה` | התקשרו פעם 2, לא ענתה |
| `לא_מגיעה` | ביטלה / הודיעה שאינה מגיעה |
| `אישרה_הגעה` | אישרה טלפונית – כרטיס QR נשלח |

**תהליך הטלפנית:**
```
פתיחת רשימת "ממתינות לאישור" לאירוע הרלוונטי (ממוינת לפי עדיפות)
      ↓
חיוג לנרשמת (ידני, מחוץ למערכת)
      ↓
תיעוד תוצאה:
  ├── אין מענה → סטטוס ניסיון 1/2 + תזמון שיחה חוזרת אוטומטי
  ├── לא מגיעה → סטטוס "לא מגיעה" + הערה אופציונלית
  └── אישרה הגעה → סטטוס "אישרה הגעה"
        ↓
        יצירת QR + שליחת WhatsApp עם כרטיס דיגיטלי אוטומטית
```

**מעקב וסטטיסטיקה לאירוע:**
```
Dashboard אירוע מציג בזמן אמת:
  ├── סך נרשמות
  ├── אישרו הגעה: X (Y%)
  ├── ניסיון ללא מענה: X
  ├── לא מגיעות: X
  └── ממתינות לאישור: X
```

**התראה אוטומטית למנהל שיווק:**
```
פעם ביום (מוגדר X ימים לפני האירוע) → חישוב אחוז אישורים
      ↓
אם (אישרו הגעה / סך נרשמות) < יעד_הקמפיין − סף_התראה:
      ↓
שליחת הודעה ל-Admin + מנהל שיווק:
  "שיעור אישורי הגעה לאירוע [שם]: [Y]%
   יעד: [Z]% | חסר: [N] אישורים
   מומלץ לשקול המשך פרסום"
```

**הגדרת יעדים (Admin לכל אירוע):**
- יעד אחוז אישורים (ברירת מחדל: 70%)
- סף התראה: מתי לשלוח (ברירת מחדל: 15% מתחת ליעד)
- ימי הפעלה: מתי מתחילים מעקב (ברירת מחדל: 7 ימים לפני האירוע)

### תהליך K – אישור חומרי פרסום *(חדש בגרסה 1.3)*

**מטרה:** ממשק מרכזי לסקירה ואישור חומרי פרסום לפני העלאה לפרסום ברשתות.

**שחקנים:** חברת פרסום (גישה חיצונית מוגבלת) + Admin

```
חברת הפרסום מעלה חומר דרך ממשק ייעודי:
  שם חומר + שיוך לקמפיין/אירוע + פלטפורמה + קובץ/URL + הערות
      ↓
Admin מקבל התראה: "חומר חדש ממתין לאישור"
      ↓
Admin בוחן החומר (מציג תצוגה מקדימה):
  ├── מאשר → חברת הפרסום מקבלת הודעה: "מאושר להעלאה" + תאריך
  └── דוחה → חברת הפרסום מקבלת הודעה + הערות לתיקון
      ↓
ארכיון: כל החומרים + סטטוס + היסטוריית סקירה שמורים במערכת
```

**שדות ישות AdvertisingMaterial:**

| שדה | תיאור |
|---|---|
| event_id | שיוך לאירוע (אופציונלי) |
| material_type | תמונה / וידאו / טקסט / קריאייטיב |
| platform | פייסבוק / אינסטגרם / Google / אחר |
| file_url | קישור לקובץ שהועלה |
| company_notes | הסבר מחברת הפרסום |
| status | `ממתין` / `מאושר` / `נדחה` |
| review_notes | פידבק מה-Admin לתיקון |
| reviewed_by | User FK (Admin) |
| uploaded_at / reviewed_at | timestamps |

**הרשאות:**
- Admin: צפייה + אישור/דחייה + עריכת הערות
- מנהל שיווק: צפייה בלבד
- חברת פרסום: העלאה + צפייה בסטטוס החומרים שלה בלבד

### תהליך L – מעקב תקציב קמפיין *(חדש בגרסה 1.3)*

**מטרה:** מעקב אחר הוצאות פרסום יומיות ומצטברות מול יעד, ישירות ממשק Meta Ads.

```
Admin / מנהל שיווק מגדיר לכל קמפיין:
  שם קמפיין + meta_campaign_id + יעד יומי + יעד כולל + תאריכי הקמפיין
      ↓
המערכת שולפת מ-Meta Ads API (פעם ביום בחצות):
  הוצאה בפועל לפי יום + חשיפות + קליקים
      ↓
תצוגה לכל קמפיין:
  ├── יעד יומי ← הוצאה בפועל היום
  ├── יעד כולל ← הוצאה מצטברת
  └── גרף ציר זמן: יעד vs. בפועל
      ↓
אם הוצאה יומית > 110% מהיעד → התראה ל-Admin + מנהל שיווק
```

**ישויות נדרשות:**

```
CampaignBudget (תקציב קמפיין)
  ├── id, event_id (FK nullable)
  ├── campaign_name, meta_campaign_id (external)
  ├── platform (facebook/instagram/google)
  ├── daily_target, total_target (DECIMAL)
  ├── start_date, end_date
  └── created_by (User FK)

CampaignSpend (הוצאה בפועל – מ-Meta API)
  ├── id, campaign_budget_id (FK)
  ├── spend_date (DATE)
  ├── actual_spend (DECIMAL)
  ├── impressions, clicks (INT)
  └── synced_at (TIMESTAMP)
```

**הרשאות:** Admin ✅ + מנהל שיווק ✅ | כל השאר ❌

## 5. מודל נתונים

### ישויות ויחסים

```
Contact (לקוחה)
  ├── id, name, phone (unique), city, family_status
  ├── source (facebook/google/landing_page/manual)
  ├── created_at, assigned_to (User FK)
  ├── status (lead_status enum)
  │
  ├── → EventRegistration (1:many)
  ├── → Donation (1:many)
  └── → ContactNote (1:many)

Event (אירוע)
  ├── id, name, date, location, description
  ├── capacity, status (draft/active/completed)
  ├── landing_page_id (FK)
  └── → EventRegistration (1:many)
      └── → EventExpense (1:many)

EventRegistration (רישום לאירוע)
  ├── id, contact_id (FK), event_id (FK)
  ├── registered_at, source (landing/manual/call_center)
  ├── barcode_token (unique, nullable – נוצר רק לאחר אישור)
  ├── barcode_sent_at (TIMESTAMP nullable)
  ├── attendance_status (registered/attended/absent)
  ├── confirmation_status (pending_call/no_answer_1/no_answer_2/declined/confirmed)
  ├── confirmed_by (User FK nullable – הטלפנית שאישרה)
  ├── confirmed_at (TIMESTAMP nullable)
  └── → Donation (1:1 optional)

Donation (תרומה)
  ├── id, contact_id (FK), event_registration_id (FK)
  ├── donated (boolean)
  ├── amount (nullable – restricted visibility)
  ├── method (cash/standing_order/credit_once/bank_transfer)
  ├── donation_date, notes
  └── recorded_by (User FK)

ContactNote (הערת טלפנית)
  ├── id, contact_id (FK)
  ├── note_text, created_at
  ├── created_by (User FK)
  └── follow_up_date (nullable)

LandingPage (דף נחיתה)
  ├── id, event_id (FK), slug (unique)
  ├── title, description, banner_image_url
  ├── primary_color, secondary_color
  ├── form_fields (JSON – שדות גמישים)
  └── published (boolean)

EventExpense (הוצאת אירוע)
  ├── id, event_id (FK)
  ├── category_id (FK), amount, description
  ├── expense_date, receipt_url
  └── created_by (User FK)

ExpenseCategory (קטגוריית הוצאה)
  └── id, name, description

InventoryItem (פריט מלאי)
  ├── id, name, description
  └── → InventoryTransaction (1:many)

InventoryTransaction (תנועת מלאי)
  ├── id, item_id (FK)
  ├── type (purchase/distribution)
  ├── quantity, transaction_date
  ├── event_id (FK – nullable, לחלוקה)
  └── created_by (User FK)

User (משתמש)
  ├── id, name, email, phone
  ├── role (admin/marketing/agent/receptionist/secretary/warehouse)
  ├── is_active (boolean)
  └── created_at

WhatsAppLog (לוג שליחות)
  ├── id, contact_id (FK), event_id (FK nullable)
  ├── message_type (registration_confirm/reminder/other)
  ├── sent_at, status (sent/failed/delivered)
  └── external_message_id

AgentShift (משמרת / נוכחות טלפנית)
  ├── id, agent_id (User FK)
  ├── shift_date (DATE)
  ├── clock_in (TIMESTAMP)
  ├── clock_out (TIMESTAMP nullable)
  ├── total_minutes (computed: clock_out - clock_in)
  ├── break_minutes (INT default 0)
  ├── net_work_minutes (computed: total - break)
  └── notes

AgentAbsence (היעדרות / חופש)
  ├── id, agent_id (User FK)
  ├── absence_date (DATE)
  ├── type (vacation/sick/holiday/unpaid/other)
  ├── approved (boolean), approved_by (User FK nullable)
  └── notes

EventConfirmationTarget (יעד אישורי הגעה לאירוע)
  ├── id, event_id (FK unique)
  ├── confirmation_target_pct (INT default 70)
  ├── alert_threshold_pct (INT default 15 – כמה % מתחת ליעד לפני התראה)
  ├── alert_days_before (INT default 7)
  └── last_alert_sent_at (TIMESTAMP nullable)

AdvertisingMaterial (חומר פרסום)
  ├── id, event_id (FK nullable)
  ├── material_type (image/video/text/creative)
  ├── platform (facebook/instagram/google/other)
  ├── file_url, company_notes
  ├── status (pending/approved/rejected)
  ├── review_notes (nullable)
  ├── reviewed_by (User FK nullable)
  ├── uploaded_at, reviewed_at (TIMESTAMP nullable)
  └── uploaded_by_email (חברת הפרסום – ללא User FK)

CampaignBudget (תקציב קמפיין)
  ├── id, event_id (FK nullable)
  ├── campaign_name, meta_campaign_id (external ID)
  ├── platform (facebook/instagram/google)
  ├── daily_target, total_target (DECIMAL)
  ├── start_date, end_date
  └── created_by (User FK)

CampaignSpend (הוצאה בפועל – מ-API)
  ├── id, campaign_budget_id (FK)
  ├── spend_date (DATE)
  ├── actual_spend (DECIMAL)
  ├── impressions, clicks (INT)
  └── synced_at (TIMESTAMP)

```

| **אבטחה** | Webhook secret token, HTTPS בלבד |
| **נדרש** | Meta Business App עם הרשאות leads_retrieval |

### 7.2 WhatsApp Business API

| פרמטר | פרטים |
|---|---|
| **סוג** | REST API – שליחת הודעות תבנית (Template Messages) |
| **ספק מומלץ** | Wati / 360dialog / Respond.io (לפי מערכת קיימת) |
| **שימושים** | אישור רישום + כרטיס דיגיטלי / תזכורת אירוע |
| **תבניות נדרשות** | 2 תבניות מאושרות על ידי Meta |
| **Fallback** | SMS דרך Twilio / אינפobip אם WhatsApp נכשל |
| **פרמטרים נדרשים** | API Key + Phone Number ID + WhatsApp Business Account ID |

### 7.3 Meta Ads API – מעקב תקציב קמפיין *(חדש גרסה 1.3)*

| פרמטר | פרטים |
|---|---|
| **סוג** | REST API – שליפת נתוני הוצאות |
| **Endpoint** | `GET /act_{ad_account_id}/insights` |
| **נתונים נשלפים** | spend, impressions, clicks לפי יום ו-campaign_id |
| **תזמון** | Cron job יומי בחצות + שליפה ידנית לפי דרישה |
| **נדרש** | Meta Business App + Marketing API permissions + Ad Account ID |
| **אבטחה** | Access Token מוצפן, HTTPS בלבד |

### 8.1 ביצועים
- עמוד מוקד טלפוני טוען תוך **< 2 שניות** עם 3,000+ לידים
- ממשק סריקת ברקוד מגיב תוך **< 0.5 שניות** לאחר סריקה
- שליחת WhatsApp מתבצעת תוך **< 30 שניות** מרישום

### 8.2 זמינות
- 99.5% uptime
- גיבוי אוטומטי יומי של מסד הנתונים
- שחזור תוך 24 שעות מכל תקלה

### 8.3 אבטחה ופרטיות
- כל התעבורה ב-HTTPS (TLS 1.3)
- סיסמאות מוצפנות (bcrypt)
- Audit Log לכל פעולה רגישה (עדכון תרומה, עדכון הרשאות, ייצוא)
- עמידה בחוק הגנת הפרטיות הישראלי
- נתוני כסף (תרומות, הוצאות) מוצפנים ב-DB
- Rate limiting על Webhook endpoints

### 8.4 רספונסיביות
- Web app: מלא ב-Desktop, חלקי ב-Tablet
- Mobile (PWA): סריקה, מלאי, הוצאות – מותאם לסמארטפון
- תמיכה ב-iOS 16+ ו-Android 10+

### 8.5 שפה ו-RTL
- ממשק בעברית מלאה
- RTL (ימין לשמאל) בכל הממשק
- פורמט תאריכים: DD/MM/YYYY

## 9. ארכיטקטורה ומחסנית טכנולוגית

### 9.1 המלצה

| שכבה | טכנולוגיה | נימוק |

| **File Storage** | Supabase Storage | קבלות, תמונות, כרטיסים דיגיטליים |
| **WhatsApp** | Wati / 360dialog SDK | ממשק REST פשוט, תבניות מנוהלות |

| **Hosting** | Vercel (Frontend) + Supabase Cloud | Auto-scaling, CI/CD מובנה |

### 13.3 דוח תרומות חודשי לטלפנית
